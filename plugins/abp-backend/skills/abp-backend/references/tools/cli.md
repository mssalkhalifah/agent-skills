# Tools — ABP CLI

> Reference for the ABP Command-Line Interface (`abp`, package id `Volo.Abp.Studio.Cli`) — the dotnet-global tool that scaffolds solutions and modules, manages ABP packages and source code, generates client proxies, switches framework channels, and bridges to ABP Studio / Kubernetes / the ABP Platform login.

## When to load this reference

- User asks how to install or update the ABP CLI, or sees errors like 'command not found: abp' or version-mismatch warnings.
- User asks 'how do I create an ABP solution from the command line' or wants the exact `abp new` flag combination for a given UI / database / mobile / tiered choice.
- User asks how to add an ABP module to an existing solution (`abp add-package`, `abp install-module`, `abp install-local-module`, `abp add-package-ref`).
- User asks how to upgrade or downgrade an ABP solution's framework version (`abp update`, `--npm`, `--nuget`, `switch-to-preview`, `switch-to-stable`, `switch-to-nightly`, `switch-to-local`).
- User asks how to generate API client proxies for Angular / C# / JavaScript (`abp generate-proxy`), or how to remove them (`abp remove-proxy`).
- User asks how to download the source code of an ABP module to customize it (`abp get-source`, `abp add-source-code`).
- User asks how to log in to abp.io from the CLI to access pro/commercial packages (`abp login`, `abp login --device`, `abp logout`, `abp login-info`).
- User asks about ABP Studio CLI vs the legacy `Volo.Abp.Cli` (the `--old` flag).
- User asks how to scaffold a module / package (`abp new-module`, `abp new-package`).
- User asks how to bundle Blazor static assets, install client libraries, or generate JWKS keys (`abp bundle`, `abp install-libs`, `abp generate-jwks`).
- User wants to recreate a solution from a saved configuration (`abp new -shi <id>`, `abp new -rcp <config.json>`).

**Audience:** Any ABP developer or DevOps engineer who scaffolds, upgrades, or maintains ABP 10.3 solutions from the terminal — including those using ABP Studio (whose CLI commands are the same `abp` binary) and CI/CD pipelines that need non-interactive `abp new` / `abp update` invocations.

## Key concepts

- ABP CLI (Volo.Abp.Studio.Cli) — dotnet global tool, binary name `abp` — the modern ABP Studio CLI; install with `dotnet tool install -g Volo.Abp.Studio.Cli`, update with `dotnet tool update -g Volo.Abp.Studio.Cli`. Reach for it for every command-line interaction with ABP 10.3.
- Legacy CLI bridge — `--old` flag (appended at end of command) and the `install-old-cli` command — invokes the older `Volo.Abp.Cli` codebase from inside the new CLI; reach for it only when a command/flag exists in the old CLI but has not been ported (e.g., `abp new ... -t console --old`).
- `abp new` — solution generator — scaffolds a startup solution from a template; the most-used CLI command. Reach for it at project kickoff or when reproducing a solution from a saved config (`-shi`, `-rcp`).
- `abp new-module` — module-template generator — creates a reusable ABP module solution (typically `module:ddd`); reach for it when starting a new ABP module/library to be packaged and consumed by other solutions.
- `abp new-package` — library-template generator — creates a single-project ABP library (e.g., `--template lib.domain`); reach for it for small reusable libraries that don't need a full module layout.
- `abp add-package` — same-solution package wiring — adds and DependsOn-wires an ABP package into the current solution; reach for it when pulling in framework packages like `Volo.Abp.AuditLogging.EntityFrameworkCore`.
- `abp install-module` / `install-local-module` — module installer — installs a published or local ABP module into the current solution and updates module dependencies/migrations; reach for it for whole modules like `Volo.Blogging`.
- `abp add-package-ref` — internal project reference — adds a project reference between two projects of the same solution; reach for it during refactoring when splitting/merging projects.
- `abp get-source` — module source-code download — downloads the source code of a (commercial) ABP module under your account into a folder so you can read or customize it; reach for it to inspect module internals or to fork.
- `abp add-source-code` — replace package with project reference — converts a NuGet module reference into a local source-code project reference inside your solution (`--add-to-solution-file`); reach for it when you need to deeply customize a commercial module.
- `abp generate-proxy` — API client-proxy generator — generates strongly-typed client proxies (`-t ng` Angular, `-t csharp`, `-t js`) by reading the running host's `/api/abp/api-definition` endpoint; the host must be running. Reach for it whenever an application service signature changes.
- `abp remove-proxy` — proxy cleanup — deletes proxies previously generated by `generate-proxy`; reach for it before regenerating after large API surface changes.
- `abp generate-razor-page` — Razor page scaffold — creates a Razor page with the ABP page-model conventions; reach for it inside MVC/Razor-Pages UI projects.
- `abp generate-jwks` — RSA JWKS key generator — produces a JSON Web Key Set used to sign OpenIddict tokens in production; reach for it once per environment when configuring OpenIddict signing/encryption certificates.
- `abp update` — bulk version updater — updates ABP NuGet and NPM packages of a project / solution / multi-solution folder to the latest matching version; flags include `--npm`, `--nuget`, `--check-all`, `--version <x.y.z>`. Reach for it for every minor/patch upgrade.
- `abp switch-to-preview` / `switch-to-stable` / `switch-to-nightly` — channel switchers — change which ABP release feed packages target; reach for them to test pre-release fixes or revert to stable.
- `abp switch-to-local` (`--paths`) — local-source switcher — swaps NuGet/NPM ABP references for project references in cloned ABP repos; reach for it when debugging into the framework itself.
- `abp clean` — bin/obj cleaner — clears compiled artifacts across the solution; reach for it when builds get stuck or after a major framework switch.
- `abp init-solution` — Studio integration — annotates an existing solution so that ABP Studio recognizes it; reach for it when adopting Studio on a hand-rolled solution.
- `abp list-templates` / `list-modules` / `list-module-sources` — discovery helpers — print the available templates, installable modules, and registered module feeds.
- `abp add-module-source` / `delete-module-source` — module-feed management — register or remove a custom modules.json feed (`-n` name, `-p` path/URL).
- `abp upgrade` — Pro edition upgrader — upgrades a Community-edition solution to a commercial edition with selected pro features (`-t app --language-management --gdpr`); requires a license.
- `abp kube-connect` / `kube-intercept` — Kubernetes integration (Business+ license) — `kube-connect` attaches the local microservice to a remote Kubernetes namespace; `kube-intercept` reroutes a service's traffic to the local process for debugging.
- `abp translate` — translation workflow — consolidates JSON localization files (`-c <culture>`), applies translations back (`--apply`), and supports machine translation via DeepL (`--online --deepl-auth-key`).
- `abp login` / `login-info` / `logout` — abp.io authentication — `login` opens a browser (or `--device` for device-code flow); credentials persist under `~/.abp/` (`%UserProfile%\.abp\` on Windows). Required to access pro/commercial packages, modules, and `get-source`/`add-source-code`.
- `abp bundle` — Blazor static-asset bundler — produces the Blazor WebAssembly/MAUI Blazor `_content` bundles and asset references; reach for it after adding/removing Blazor UI packages.
- `abp install-libs` — NPM client-side installer — installs MVC/Blazor client-side libraries declared in the project's `abp.resourcemapping.js` / `libman.json`; reach for it when client-side files appear missing after a fresh clone.
- `abp cli update` / `cli check-version` / `cli clear-cache` — self-management — updates, version-checks, or clears caches of the CLI itself.
- `abp clear-download-cache` — template-cache purge — deletes cached template downloads under `~/.abp/`; reach for it when `abp new` keeps using a stale template.
- `abp mcp-studio` — Model Context Protocol bridge — exposes ABP Studio operations as an MCP server for AI assistants.
- Global flags — `--skip-cli-version-check` / `-scvc`, `--skip-extension-version-check` / `-sevc`, `--old`, `--help` / `-h` — apply to most commands; `-scvc` is what you reach for in CI to avoid network round-trips.
- Solution config storage — `cli-config.json` / `abp.studio.json` under `~/.abp/` (Windows: `%UserProfile%\.abp\`) — stores login tokens, solution-history records (consumed by `abp new -shi`), and CLI preferences.

## Configuration pattern

Unlike runtime ABP modules, the CLI is configured at install time and per-invocation via flags — not through `AbpModule.ConfigureServices`. There are three configuration surfaces:

1) **Install / update the CLI** (one-time, per machine):
```bash
# Install (modern CLI)
dotnet tool install -g Volo.Abp.Studio.Cli

# Update
dotnet tool update -g Volo.Abp.Studio.Cli

# Verify
abp --version

# Optional: install legacy CLI for `--old` fallback
abp install-old-cli
```

2) **Authenticate against abp.io** (once per machine, then refreshed silently):
```bash
abp login                # browser flow
abp login --device       # device-code flow (CI / headless)
abp login-info           # show current identity
abp logout               # clear credentials
```
Login tokens are written under `~/.abp/` (Windows: `%UserProfile%\.abp\`) and are required for `get-source`, `add-source-code`, and pro module installs.

3) **Per-invocation flags** — every command takes its own options plus four global flags:
- `--skip-cli-version-check` / `-scvc` — recommended in CI to avoid version checks.
- `--skip-extension-version-check` / `-sevc` — same idea for Studio extensions.
- `--old` — append at the end of any command to invoke the legacy `Volo.Abp.Cli` implementation.
- `--help` / `-h` — per-command help (`abp new --help`).

There is no `services.AddAbpCli(...)` because the CLI is a process, not a runtime module; conversely, anything the CLI scaffolds (modules, solutions, proxies) is consumed by the application's `AbpModule` classes in the normal way.

## Code examples

### Install or update the ABP CLI and log in

_First-time setup of the CLI on a developer machine, including login to access commercial packages._

```bash
# Install once
dotnet tool install -g Volo.Abp.Studio.Cli

# Or update an existing install
dotnet tool update -g Volo.Abp.Studio.Cli

# Authenticate (browser flow)
abp login

# Or for headless / CI machines
abp login --device

# Verify
abp login-info
```

**Key lines:** `Volo.Abp.Studio.Cli` is the package id for the ABP 10.3 CLI; the binary it installs is named `abp`. `abp login` is what unlocks pro features (`get-source`, `add-source-code`, commercial modules). `--device` returns a code to type at https://abp.io/account/device, which is the only viable flow in CI/CD or remote-SSH sessions.

### Scaffold a Layered ABP solution with Angular UI and PostgreSQL

_Creating a new layered solution with the most common production combination — Angular front end, EF Core + PostgreSQL, no mobile, in its own folder._

```bash
abp new Acme.BookStore \
  -t app \
  -u angular \
  -m none \
  --database-provider ef \
  --database-management-system PostgreSQL \
  --separate-auth-server \
  -csf
```

**Key lines:** `-t app` chooses the Layered template; `-u angular` adds an Angular SPA project; `-m none` skips React Native/MAUI; `--database-provider ef` selects EF Core; `--database-management-system PostgreSQL` swaps the default SQL Server for PostgreSQL connection strings/providers; `--separate-auth-server` splits AuthServer from the API host (production-ready for OpenIddict); `-csf` (`--create-solution-folder`) places everything in `Acme.BookStore/` rather than the current dir.

### Generate Angular client proxies after an API change

_Re-generating Angular service proxies after adding a new application-service method on a running backend._

```bash
# 1. Start the backend (HttpApi.Host) so it serves /api/abp/api-definition
#    e.g. dotnet run --project src/Acme.BookStore.HttpApi.Host

# 2. From the Angular project root
cd angular
abp generate-proxy -t ng -url https://localhost:44302/

# Other client types use the same -url with a different -t:
# abp generate-proxy -t csharp -url https://localhost:44302/   # Blazor / MAUI
# abp generate-proxy -t js     -url https://localhost:44302/   # plain JS / RN
```

**Key lines:** `generate-proxy` reads the JSON returned by `/api/abp/api-definition` on the running host, so the host must be running and reachable. `-t ng` writes TypeScript service-proxies into `src/app/proxy/`; `-t csharp` produces C# clients used by Blazor/MAUI; `-t js` produces JS proxies used by React Native and legacy MVC. If you change the URL/port (e.g., when running behind a separate AuthServer), update the `-url`. (UI-side consumption of the generated proxies is covered in the abp-frontend skill.)

### Add a published module and bring its source code in-tree

_Pulling Volo.Blogging into an existing solution as a NuGet module, then converting it to local source for customization._

```bash
# Add as NuGet packages (default)
abp install-module Volo.Blogging

# Later, when you need to customize internals,
# replace the NuGet refs with a local copy of the module's source
abp get-source Volo.Blogging --version 10.3
abp add-source-code Volo.Blogging --add-to-solution-file
```

**Key lines:** `install-module` modifies the right `*.Module.cs` files to add `[DependsOn(typeof(BloggingModule))]`, registers EF Core migrations, and updates `appsettings.json` connection strings. `get-source` only downloads source under `~/abp/get-source/`; `add-source-code` is what actually rewires the solution to project-reference the local copy (and adds the projects to the .sln). Both require `abp login` because Volo.Blogging is a commercial module.

### Upgrade an ABP solution from 10.2 to 10.3

_Updating every ABP NuGet package and Angular @abp/* npm package in a multi-project solution to ABP 10.3._

```bash
# Update both NuGet and NPM (default)
abp update

# Or split the operation:
abp update --nuget          # NuGet packages only
abp update --npm            # NPM packages only

# Pin to a specific version
abp update --version 10.3.0

# In CI, skip the network version check
abp update --version 10.3.0 -scvc
```

**Key lines:** Run `abp update` from the solution root (or a folder containing multiple solutions). It walks every `.csproj` for `Volo.Abp.*` package refs and every `package.json` for `@abp/*` entries, bumping each to the latest (or `--version`-pinned) match. Use `--npm` / `--nuget` to run them separately when you want to validate the .NET upgrade before touching Angular. `--check-all` validates each package independently, useful for diagnosing version drift between commercial and community packages.

### Recreate a solution non-interactively in CI

_Reproducing a known-good solution layout in CI from a checked-in JSON config — useful for regenerating sample apps, demos, or template tests._

```bash
# Save a config once (e.g., from `abp new` Studio output) at MyTests/config.json,
# then in CI:
abp new -rcp MyTests/config.json -scvc

# Or replay a previous local-history entry by id
abp new -shi dbb1afa9-190e-419a-842d-2780bb1bad1f -scvc
```

**Key lines:** `-rcp` (ready-config-path) reads a JSON file with the same options that the interactive Studio wizard would write — UI, db, tiered, modules, etc. `-shi` (solution-history-id) replays an entry from `~/.abp/`'s solution-history store, which `abp new` writes after every successful generation. `-scvc` (`--skip-cli-version-check`) is critical in CI to avoid extra network calls and stalled jobs.

## Common mistakes

- Mistake: Using the wrong package id (`Volo.Abp.Cli`) when installing on ABP 10.3. Why it's wrong: `Volo.Abp.Cli` is the legacy CLI; ABP 10.3 docs and Studio assume the new `Volo.Abp.Studio.Cli`, and many commands (`mcp-studio`, `kube-connect`, `init-solution`, newer `add-source-code` flags) only exist there. Correct pattern: `dotnet tool install -g Volo.Abp.Studio.Cli`. Use the `--old` suffix on individual commands if you specifically need legacy behavior, e.g., `abp new Acme.X -t console --old`.
- Mistake: Running `abp generate-proxy` against a backend that isn't running. Why it's wrong: the command reads the live `/api/abp/api-definition` endpoint over HTTP; if the host is stopped or behind a redirect, the call fails or generates an empty/partial proxy. Correct pattern: start the HttpApi.Host (or whichever project owns `/api/abp/api-definition`), confirm in a browser that the URL returns JSON, then run `abp generate-proxy -t ng -url https://localhost:44302/` with the exact running URL.
- Mistake: Forgetting `abp login` before `abp install-module Volo.Foo`, `abp get-source`, or `abp add-source-code` for a commercial module. Why it's wrong: the CLI silently downloads only public packages, then fails when it can't resolve the commercial NuGet feed, often with confusing 401/403 errors. Correct pattern: `abp login` (or `abp login --device` in CI) once, verify with `abp login-info`, then run the install/source command. In CI, store the login state under `~/.abp/` as a cached artifact between runs.
- Mistake: Running `abp update` from the wrong directory and getting partial upgrades. Why it's wrong: `abp update` walks down from the current folder; running it inside `src/MyProject.Web/` will only update that project's `.csproj`, leaving Domain/Application/Tests on the old version. Correct pattern: cd to the solution root (the folder containing the `.sln` and the `angular/` sibling) and run `abp update` there; for monorepos with multiple solutions, run it one level higher and let it walk every solution.
- Mistake: Wrapping `--connection-string` without quotes (or with single quotes on Windows PowerShell). Why it's wrong: connection strings contain `;` and `=`, which the shell will split or reinterpret, and the resulting `appsettings.json` will be malformed or truncated. Correct pattern (cross-shell-safe): `--connection-string "Server=localhost;Database=BookStore;Trusted_Connection=True"`. On PowerShell prefer escaped double quotes; on bash use single outer quotes if the value contains no single quotes itself.
- Mistake: Using `--skip-migrations` without `--skip-migrator` (or vice-versa). Why it's wrong: `--skip-migrations` removes the EF Core migration files but the solution still ships a `.DbMigrator` project that expects them, leading to startup errors when DbMigrator runs. Correct pattern: pass both together — `--skip-migrations --skip-migrator` — when you intend to manage migrations outside ABP; otherwise pass neither.
- Mistake: Treating `abp new -shi <id>` as portable across machines. Why it's wrong: solution-history ids live in `~/.abp/`'s local store; an id from another developer's machine resolves to nothing on yours. Correct pattern: for portable, repeatable scaffolding, save a JSON config file and use `abp new -rcp <path-to-config.json>` instead — that file can be checked into a repo.
- Mistake: Calling `abp switch-to-preview` / `switch-to-nightly` on a solution then committing without a `--version` pin. Why it's wrong: every subsequent `dotnet restore` may pull a newer pre-release, breaking reproducibility. Correct pattern: after switching channels, run `abp update --version 10.3.0-rc.X` to pin a specific version, or revert with `abp switch-to-stable` before merging.
## Version pins (ABP 10.3)

- ABP 10.3 ships `Volo.Abp.Studio.Cli` as the canonical CLI package; the older `Volo.Abp.Cli` is reachable only via the `--old` flag and the `install-old-cli` command. Future major versions may drop `--old`.
- `abp new -t microservice` and `abp kube-connect` / `kube-intercept` require a Business or higher license in 10.3; running them on a Community login fails at command parse / token validation time.
- `abp generate-proxy` URL endpoint is `/api/abp/api-definition`; `-t` accepts `ng`, `csharp`, `js` in 10.3 — additional client types may appear in later versions.
- Login tokens and solution-history are stored under `~/.abp/` on macOS/Linux and `%UserProfile%\.abp\` on Windows; the exact filenames inside (e.g., `cli-config.json`, `abp.studio.json`) are implementation details that have changed between releases. [uncertain] — official docs do not exhaustively enumerate the file layout for 10.3.
- Global flag aliases in 10.3: `--skip-cli-version-check` = `-scvc`, `--skip-extension-version-check` = `-sevc`. The shorter `-s` form has not been stable across versions and should be avoided in scripts.
- `abp upgrade -t app --language-management --gdpr` is the documented Pro-edition upgrade path in 10.3; the exact set of opt-in flags (`--gdpr`, `--language-management`, etc.) tracks the available Pro modules and may grow per release.
- `abp mcp-studio` (Model Context Protocol bridge) is new in the 10.x line; its protocol surface should be considered experimental and may change between minor versions. [uncertain]
- `-csf` (`--create-solution-folder`) defaults to false; `abp new` without it scaffolds into the current directory, which often surprises users coming from `dotnet new`. This default has been stable through 10.3 but is worth pinning explicitly in scripts.
