# Code Review: `OrderController` (ABP 10.3 Layered)

This controller violates several ABP Layered conventions. The biggest issue is that **all orchestration, authorization, and persistence happen in the controller** — the work belongs in an `OrderAppService` in the `Application` project. The controller (in the `HttpApi` project) should be a thin pass-through, or removed entirely in favor of ABP's auto API controllers.

## Issues

### 1. Wrong layer for orchestration
The controller injects `IRepository<Order, Guid>`, `OrderManager`, and `IObjectMapper`, then drives the full use case: build an aggregate via the manager, insert via the repository, map to DTO. That orchestration is the textbook responsibility of an **Application Service**, not the HTTP layer.

`abp-dependency-rules` is explicit:

> HttpApi: Can Reference → Contracts only
> Application: Can Reference → Domain, Contracts

A controller in `HttpApi` should only depend on `*.Application.Contracts` (i.e., `IOrderAppService` / `OrderDto` / `CreateOrderDto`). Injecting `IRepository<Order, Guid>` and `OrderManager` forces `HttpApi` to reference the `Domain` project, which breaks the layered direction.

From `abp-dependency-rules` ("Always Do"):

```csharp
// HttpApi depending on contracts only
public class BookController : AbpController
{
    private readonly IBookAppService _bookAppService; // CORRECT
}
```

### 2. Imperative role check instead of declarative permission
```csharp
if (!_currentUser.IsInRole("sales"))
{
    throw new AbpAuthorizationException();
}
```
Two problems:

- **Role string, not permission.** ABP's authorization model is permission-based; roles are how permissions get granted. Checking a hard-coded role name skips the whole `PermissionDefinitionProvider` pipeline (no UI toggling, no per-tenant overrides, no audit of what "sales" means).
- **Imperative when it should be declarative.** `abp-authorization` shows the canonical pattern is `[Authorize(BookStorePermissions.Books.Create)]` on the app service method:

  ```csharp
  [Authorize(BookStorePermissions.Books.Create)]
  public virtual async Task<BookDto> CreateAsync(CreateBookDto input) { ... }
  ```

Define `OrderPermissions.Create` in `*.Application.Contracts`, register it in a `PermissionDefinitionProvider`, and apply `[Authorize(OrderPermissions.Create)]` at the **AppService layer**. The class-level `[Authorize]` on the controller stays for "must be logged in," but specific permission checks belong on the service method.

### 3. Injecting services already on the base class
`AbpController` already exposes `CurrentUser`, `GuidGenerator`, `ObjectMapper`, `L`, etc. from `abp-core`:

> `L` (StringLocalizer), `AuthorizationService`, `FeatureChecker` — available on `ApplicationService`, `AbpController`.

And from `abp-application-layer` best practices:

> Use base class properties (`Clock`, `CurrentUser`, `GuidGenerator`, `L`) instead of injecting these services.

So `ICurrentUser` and `IObjectMapper` constructor parameters are noise — drop them and use `CurrentUser` / `ObjectMapper` from the base class (and only in the app service, not the controller, per issue #1).

### 4. `IObjectMapper` use in a Mapperly solution
ABP 10.3 templates default to **Mapperly** (compile-time), not AutoMapper. From `abp-application-layer`:

> ABP supports both Mapperly and AutoMapper integrations. But the default mapping library is Mapperly. You need to first check the project's active mapping library.

If the solution is Mapperly-based, inject `OrderMapper` and call `_orderMapper.MapToDto(order)` instead of the runtime `IObjectMapper`. If it's an AutoMapper solution, the current call style is fine — but verify.

### 5. `autoSave: true` is a smell here
Application services run inside an ambient unit of work; the UoW commit handles the save. Passing `autoSave: true` on `InsertAsync` is only needed when you need the inserted entity's identity values back **inside the same method** (e.g., to read `Id` produced by the DB). Since `Id` is generated by `GuidGenerator.Create()` and passed to `OrderManager.CreateAsync`, `autoSave` shouldn't be necessary — let the UoW flush at the end. Drop it unless there's a documented reason.

### 6. Method-name anti-pattern is OK here, but watch the DTO
`CreateAsync(CreateOrderDto input)` is fine (no entity name in method — good, per `abp-application-layer`). Just confirm `CreateOrderDto` and `OrderDto` live in `*.Application.Contracts`, not in `HttpApi` or `Application`.

### 7. Consider deleting the controller entirely
ABP auto-generates API controllers for any `IApplicationService`. From `abp-application-layer`:

> ABP automatically generates API controllers for application services:
> - Interface must inherit `IApplicationService` (which already has `[RemoteService]` attribute)
> - HTTP methods determined by method name prefix (Get, Create, Update, Delete)

Unless you need a non-conventional route, a custom binding, or special HTTP semantics, you can delete `OrderController` and let `IOrderAppService.CreateAsync` be exposed automatically at `POST /api/app/order`. Custom controllers are an opt-out, not the default.

## Suggested Refactor

**`*.Application.Contracts/Orders/IOrderAppService.cs`**
```csharp
public interface IOrderAppService : IApplicationService
{
    Task<OrderDto> CreateAsync(CreateOrderDto input);
}
```

**`*.Application.Contracts/Permissions/OrderPermissions.cs`**
```csharp
public static class OrderPermissions
{
    public const string GroupName = "Ordering";

    public static class Orders
    {
        public const string Default = GroupName + ".Orders";
        public const string Create  = Default + ".Create";
    }
}
```
(Registered in an `OrderingPermissionDefinitionProvider` — see `abp-authorization`.)

**`*.Application/Orders/OrderAppService.cs`**
```csharp
public class OrderAppService : ApplicationService, IOrderAppService
{
    private readonly IRepository<Order, Guid> _orderRepository;
    private readonly OrderManager _orderManager;
    private readonly OrderMapper _orderMapper; // Mapperly; or use ObjectMapper if AutoMapper

    public OrderAppService(
        IRepository<Order, Guid> orderRepository,
        OrderManager orderManager,
        OrderMapper orderMapper)
    {
        _orderRepository = orderRepository;
        _orderManager = orderManager;
        _orderMapper = orderMapper;
    }

    [Authorize(OrderPermissions.Orders.Create)]
    public virtual async Task<OrderDto> CreateAsync(CreateOrderDto input)
    {
        var order = await _orderManager.CreateAsync(
            GuidGenerator.Create(),   // from base class
            input.CustomerId,
            input.Number);

        await _orderRepository.InsertAsync(order);
        return _orderMapper.MapToDto(order);
    }
}
```

**`*.HttpApi/Controllers/OrderController.cs`** (only if you really need a custom controller; otherwise delete and rely on auto API):
```csharp
[Authorize]
[Route("api/orders")]
public class OrderController : AbpController
{
    private readonly IOrderAppService _orderAppService;

    public OrderController(IOrderAppService orderAppService)
    {
        _orderAppService = orderAppService;
    }

    [HttpPost]
    public Task<OrderDto> CreateAsync(CreateOrderDto input)
        => _orderAppService.CreateAsync(input);
}
```

## Summary of Changes
- Move orchestration (manager call, repository insert, mapping) out of the controller and into `OrderAppService` (`abp-application-layer`, `abp-dependency-rules`).
- Replace `_currentUser.IsInRole("sales")` with declarative `[Authorize(OrderPermissions.Orders.Create)]` on the app service method; define the permission in a `PermissionDefinitionProvider` (`abp-authorization`).
- Drop injected `ICurrentUser`, `IObjectMapper`, and `GuidGenerator` references — use the base class properties (`abp-core`, `abp-application-layer`).
- Use the Mapperly mapper (`OrderMapper`) if the solution is Mapperly-based (default in 10.3), else `ObjectMapper` (`abp-application-layer`).
- Drop `autoSave: true` unless explicitly needed; let the ambient UoW flush.
- Consider removing `OrderController` and relying on ABP's auto API controllers; keep it only if you need the custom `api/orders` route.
